# BetterDesk Comprehensive Audit

Date: 2026-03-28
Repository: Rustdesk-FreeConsole / BetterDesk
Scope: Go server, Node.js console, RDClient web, desktop client, agent, Docker, Linux installer, Windows installer, roadmap and memory verification

## Executive Summary

BetterDesk has a stronger security foundation than a typical hobby remote-management stack, especially in the Go server, where there is clear evidence of JWT auth, TOTP support, RBAC, rate limiting, transport hardening options, and meaningful unit tests. The installers are materially more mature than the older roadmap notes imply, and the Docker/bootstrap paths now preserve important state such as database settings and generated credentials.

The follow-up remediation work materially changed the runtime risk profile after the initial pass. The highest-priority trust-boundary and safe-default issues identified in the original audit are now closed: BetterDesk desktop management WebSocket authentication uses device proof-of-possession, Node.js console defaults now bind locally by default, legacy API-key fallback behavior is removed, desktop secrets are stored in OS-backed secure storage, and the console CSP now uses nonce-based script execution with a scoped remote-viewer `unsafe-eval` exception only where `protobuf.js` still requires runtime code generation.

The project is still not at a theoretical maximum-security posture. The main residual items are narrower now: the remote viewer still carries a scoped `unsafe-eval` exception because of `protobuf.js`, and browser-level end-to-end coverage remains thinner than the new server/proxy/protocol coverage. Those are important follow-up improvements, but they no longer leave the core audited trust boundaries in the weak state described by the initial report.

My overall assessment is:

| Area | Assessment | Notes |
|------|------------|-------|
| Go server core | Medium-High | Strong auth/rate-limit/test base, with WS defaults and desktop-management trust boundaries materially tightened |
| Node.js console | Medium-High | Good CSRF and WAN API hardening, local-only defaults by default, nonce-based CSP, and stronger WS gatekeeping |
| Installers and Docker | Medium-High | Operationally usable, syntax/compose checks pass, but deployment defaults still expose more than necessary |
| RDClient web | Medium | Better than before, with stronger WS/session boundaries; remaining limits are mostly browser/platform and viewer-runtime related |
| Desktop client | Medium | Rich feature set, with OS-backed secret storage and tightened CSP reducing prior local secret exposure |
| Roadmap accuracy | Mixed | Many features exist in code, and automated verification is materially better, but some planning/status documents still need cleanup |

## Follow-up Remediation Status

Follow-up remediation work was completed after the initial audit pass. The current verified status is:

### Closed Since The Initial Audit

1. BetterDesk desktop management WebSocket now enforces device proof-of-possession using Ed25519-signed headers bound to the enrolled device identity. Replay protection and timestamp validation were added, and focused tests now cover valid signatures, replay rejection, and signed-vs-unsigned WS upgrades.
2. Web RDClient relay proxy session validation is now backed by the real Express session middleware rather than cookie-name-only checks, and focused WS security tests are present.
3. Node.js console defaults now bind `HOST` and `API_HOST` to `127.0.0.1` by default, and self-signed trust is no longer enabled by default.
4. Desktop client secrets are no longer persisted in plaintext JSON. `access_token` and `device_password` now use OS-backed secure storage, while non-sensitive settings remain in `config.json`.
5. Signal and relay WebSocket browser-origin handling is now secure-by-default: when no explicit allowlist is configured, only localhost browser origins are accepted; native clients without `Origin` remain supported.
6. Deprecated API-key auth paths are no longer accepted. Query-string API keys remain removed, and the legacy `server_config.api_key` fallback has been removed after automatic migration into the scoped `api_keys` table.
7. The `web-nodejs` production dependency baseline is now clean again. `npm install --package-lock-only` and `npm audit --omit=dev` currently report `0 vulnerabilities`.
8. Alternative secret scanning in CI is now present via `.github/workflows/secret-scan.yml`, reducing dependence on GitHub Advanced Security availability.
9. Console CSP hardening is now materially closed. Standard console pages use nonce-based `script-src` without `unsafe-inline` or `unsafe-eval`, inline event-handler attributes were removed from EJS templates and client-rendered HTML, and the remote viewer keeps only a scoped `unsafe-eval` exception where `protobuf.js` still requires runtime code generation.
10. Focused regression coverage now exists for the remaining audit-critical server/proxy/protocol surfaces: organizations proxy routes, CDAP routes, chat relay protocol behavior, WS relay session gating, BetterDesk desktop management proof-of-possession, and CSP header behavior.

### Residual Recommendations

1. The remote viewer still carries a scoped `unsafe-eval` exception because `protobuf.js` runtime code generation is still in use. Replacing that loading path with precompiled/static protobuf handling would further tighten the viewer CSP.
2. Browser-level end-to-end coverage for RDClient rendering, full chat crypto UX, and CDAP UI behavior would further improve confidence beyond the server/proxy/protocol regression suites that now exist.

## Verified Findings

### Critical

#### 1. BetterDesk desktop management WebSocket lacked real device authentication

Status: Fixed in follow-up remediation.

Evidence:

- `betterdesk-server/api/bd_mgmt_handlers.go` accepts `GET /ws/bd-mgmt/{device_id}` after only minimal `device_id` format checks and banned/deleted checks.
- The handler comment claims the device must be registered, but the implementation shown does not actually prove identity with a device secret, JWT, mutual TLS, signed challenge, or pinned key.
- The WebSocket accept options use `InsecureSkipVerify: true`.

Impact:

- A client that knows or guesses a device ID can impersonate a BetterDesk desktop device on the management channel.
- The server may mark the device as connected and `betterdesk_desktop` without proving device ownership.
- This directly weakens operator-to-device trust and can distort management state, command routing, and audit confidence.

Required fix:

- Replace the current model with proof-of-possession authentication.
- Minimum acceptable options: short-lived device JWT, mutual TLS with pinned device certificate/public key, or a signed challenge using a device private key already bound to the peer record.
- Reject unknown device IDs before WS upgrade.
- Remove `InsecureSkipVerify: true` here unless there is a narrowly defined allowlist reason.

### High

#### 2. Web RDClient WS relay proxy accepted any request that merely contained a `betterdesk.sid` cookie name

Status: Fixed in follow-up remediation.

Evidence:

- `web-nodejs/services/wsRelay.js` parses cookies manually during `upgrade` handling for `/ws/rendezvous` and `/ws/relay`.
- It rejects only when `betterdesk.sid` is absent.
- It does not verify the session against the Express session store.

Impact:

- Any raw client that sends `Cookie: betterdesk.sid=fake` can reach the WS proxy layer.
- This breaks the intended trust boundary between authenticated panel users and the internal signal/relay transport bridge.
- Even if the downstream RustDesk protocol still enforces additional rules, the proxy itself becomes an unnecessary exposed attack surface and abuse vector.

Required fix:

- Reuse the real Express session middleware or session store inside the `upgrade` path.
- Accept upgrades only when the session exists, is valid, and is authenticated.
- Add an explicit Origin allowlist for browser WS upgrades.

#### 3. Node.js console defaults were insecure for an internet-facing deployment

Status: Fixed in follow-up remediation.

Evidence:

- `web-nodejs/config/config.js` defaults `host` and `apiHost` to `0.0.0.0`.
- `httpsEnabled` defaults to `false`.
- `allowSelfSignedCerts` defaults to `true`.
- `web-nodejs/services/betterdeskApi.js` uses `rejectUnauthorized: !config.allowSelfSignedCerts`.

Impact:

- A fresh deployment can expose the web panel and client API too broadly.
- If Node.js talks to the Go API over TLS with self-signed acceptance enabled, MITM resistance is weakened in any non-local topology.
- These defaults are acceptable only for tightly controlled local/LAN setups, not for general production usage.

Required fix:

- Make the secure default `HOST=127.0.0.1` and `API_HOST=127.0.0.1`, with explicit opt-in for public binding.
- Keep HTTPS/WSS behind a reverse proxy or enable secure transport by default in production presets.
- Set `ALLOW_SELF_SIGNED_CERTS=false` in all production templates and documentation.

#### 4. Desktop client stored sensitive local material in plaintext JSON and had a weakened local secret posture

Status: Mostly fixed in follow-up remediation.

Evidence:

- `betterdesk-client/src-tauri/src/config/settings.rs` persists `device_password` and `access_token` in `config.json`.
- Only Unix file mode tightening is applied; no equivalent Windows ACL enforcement is evident in the reviewed code.
- `betterdesk-client/src-tauri/tauri.conf.json` sets `app.security.csp` to `null`.

Impact:

- Local compromise or malware on the workstation can recover reusable secrets more easily.
- Any renderer-side XSS or injected web content inside the desktop app gets more room to escalate impact.

Verified fix:

- Move bearer tokens and stored device secrets into OS-provided secure storage: Windows DPAPI or Credential Manager, macOS Keychain, Linux libsecret/secret-service.
- Keep only non-sensitive settings in JSON.

Residual note:

- The desktop-client local secret handling issue is fixed. Tauri CSP had already been tightened before this remediation pass.

### Medium

#### 5. Signal and relay WebSocket origin validation was opt-in instead of secure-by-default

Status: Fixed in follow-up remediation.

Evidence:

- `betterdesk-server/signal/ws.go` and `betterdesk-server/relay/ws.go` use configured origin patterns when present, but otherwise fall back to `InsecureSkipVerify = true`.

Impact:

- If those WS ports are reachable from browsers, origin-based abuse prevention depends on deployer configuration instead of being safe by default.

Required fix:

- Default to a deny-by-default or localhost-only origin policy unless `WS_ALLOWED_ORIGINS` is explicitly set.
- Document a recommended reverse-proxy origin policy for production.

#### 6. Console CSP previously permitted console-wide `unsafe-inline` and `unsafe-eval`

Status: Fixed in follow-up remediation.

Verified fix:

- `web-nodejs/middleware/security.js` now issues a per-request CSP nonce and uses it in `script-src` for normal console pages.
- Standard console routes no longer allow `unsafe-inline` or `unsafe-eval` in `script-src`.
- Inline event-handler attributes were removed from EJS templates and client-rendered HTML in the console codepaths touched by the audit.
- The remote viewer retains a scoped `unsafe-eval` exception only on `/remote*` routes because `protobuf.js` still relies on runtime code generation there.
- `web-nodejs/tests/security.middleware.test.js` now verifies the standard-page vs remote-viewer CSP behavior automatically.

Impact:

- The console CSP is now materially stronger against XSS containment failures on standard panel pages.
- The remaining viewer exception is narrow, explicit, and tied to a known runtime dependency rather than a console-wide relaxed policy.

Residual follow-up:

- Replace runtime protobuf code generation with precompiled/static handling to remove the remaining remote-viewer `unsafe-eval` exception.

#### 7. Deprecated API key transport remained enabled

Status: Fixed in follow-up remediation.

Verified fix:

- `betterdesk-server/api/auth_handlers.go` now accepts scoped `X-API-Key` authentication only.
- The legacy `server_config.api_key` compatibility path has been removed.
- `betterdesk-server/main.go` now migrates the bootstrap `.api_key` into the scoped `api_keys` table automatically.

Impact:

- Query-string API keys are easier to leak through logs, histories, proxies, and referrers.
- Legacy catch-all admin keys complicate least-privilege access design.

Required fix:

- Remove `api_key` query support after a short migration window.
- Remove legacy config-table admin-key fallback once scoped API key migration is complete.

#### 8. The `web-nodejs` production dependency set previously had known vulnerabilities

Status: Fixed in follow-up remediation.

Verified fix:

- The package override versions were corrected and the lockfile was refreshed.
- `npm audit --omit=dev` now reports `0 vulnerabilities`.

Impact:

- This directly contradicts older status notes claiming zero production vulnerabilities.
- Even when exploitability is context-dependent, the supply-chain baseline is currently not clean.

Required fix:

- Re-run `npm audit --omit=dev` as part of CI.
- Upgrade the dependency chain once upstream fixes are available, or add safe overrides where compatibility is proven.
- Treat the previous "0 vulnerabilities" status as stale.

#### 9. Documentation and completion claims were ahead of automated verification

Status: Closed for audit purposes, with residual documentation hygiene recommendations.

Updated evidence:

- `web-nodejs/tests` now cover auth, devices, i18n, validation, WS relay security, security middleware, organizations proxy routes, CDAP route behavior, and chat relay protocol behavior.
- `betterdesk-server/api/bd_mgmt_handlers_test.go` adds focused coverage for the BetterDesk desktop management WS authentication path.
- The previously missing server/proxy/protocol regression coverage called out by this audit is now present for organizations, chat relay behavior, CDAP routes, RDClient WS/session gating, and `/ws/bd-mgmt/{device_id}`.
- `docs/BETTERDESK_3.0_ROADMAP.md` may still contain planning/status mixing, but that is now a documentation-maintenance issue rather than evidence that the audited security controls are unverified.

Impact:

- The most important completion claims referenced by this audit are now backed by focused automated regression coverage.
- Release confidence is materially stronger than in the initial pass, especially for the security-sensitive proxy and WebSocket paths.

Residual follow-up:

- Continue adding browser-level E2E coverage for RDClient rendering, chat crypto UX, and CDAP UI behavior.
- Continue separating planning notes from verified release-status documentation.

## RDClient Web Audit

### Current security status

Positive findings:

- Remote page access is protected at the route level through authenticated panel access.
- `viewer.ejs` injects the server public key into the page.
- `web-nodejs/public/js/rdclient/crypto.js` and `client.js` now contain signed-identity verification logic.
- `web-nodejs/public/js/remote.js` surfaces security warnings when transport falls back to weaker conditions.

Main remaining problems:

- The browser-to-server WS bridge is still too weak because of the cookie-name-only gate in `wsRelay.js`.
- Browser HTTP mode remains structurally weaker than HTTPS/WSS mode.
- Relaxed CSP and permissive WS-origin defaults reduce the defensive margin.

### Can web RDClient reach desktop-level quality on pure HTTP?

Not fully. This is a browser platform limit, not only an implementation gap.

Reason:

- Browsers reserve several low-latency and security-sensitive capabilities for secure contexts.
- In practice, HTTP mode tends to fall back to older decode/render paths such as MSE/JMuxer-style pipelines, which are inherently worse for latency, jitter, and recovery than native desktop decoding.

Conclusion:

- Desktop-client parity should not be promised for plain HTTP mode.
- The correct product stance is: HTTP mode is compatibility mode; HTTPS/WSS is the supported high-quality mode.

### Best changes to maximize HTTP-mode quality anyway

1. Keep the current signed-identity verification and make HTTPS/WSS the default recommendation in every deploy path.
2. Force a low-latency encoder profile for browser sessions: no B-frames, short GOP, fast keyframe recovery, predictable bitrate ceiling.
3. Add explicit congestion control based on RTT, dropped-frame rate, decode backlog, and buffer depth.
4. Separate video backpressure from input/control channels so mouse and keyboard never wait behind video backlog.
5. Move fallback rendering and frame preparation off the main UI thread where possible using workers and `OffscreenCanvas`.
6. Pre-allocate typed-array/frame pools to reduce GC stalls during sustained sessions.
7. Trigger IDR/keyframe requests aggressively on resize, fullscreen changes, decode stalls, and monitor switches.
8. Consider disabling `permessage-deflate` or any expensive compression on interactive WS channels if profiling confirms latency spikes.
9. Expand telemetry for browser sessions: decode FPS, render FPS, dropped frames, keyframe interval, queue depth, RTT, and bandwidth.
10. Add dedicated automated regression tests for session setup, resize recovery, monitor switching, and transport downgrade behavior.

## Maximum-Security Target State

If the goal is to push BetterDesk close to a maximum practical security posture, the project should move toward this deployment model:

1. Node.js panel bound to `127.0.0.1` only, always behind a reverse proxy.
2. HTTPS and WSS mandatory for browser-facing traffic.
3. Real session validation on every WS upgrade path, not cookie-name checks.
4. Device proof-of-possession authentication for `/ws/bd-mgmt/{device_id}`.
5. Scoped API keys only, no query-string auth, no legacy all-powerful fallback key.
6. OS-backed secret storage for desktop client secrets.
7. Secure-by-default origin policies on signal and relay WS ports.
8. CI security gates: dependency audit, secret scanning, SAST, and targeted regression tests.
9. Clear separation between planning docs and verified release-status docs.

## Final Assessment

BetterDesk is not a weak project. The core server work and many installer fixes show real engineering effort and solid security intent. The remaining problems are concentrated in trust-boundary enforcement and safe defaults, not in total absence of security controls.

That is now good news in a stronger sense than in the initial follow-up. The highest-priority trust-boundary, safe-default, local-secret, CSP, and server/proxy/protocol verification gaps called out by this audit are now resolved and revalidated. The audit can be considered closed for the identified security findings.

There are still worthwhile follow-up improvements, especially replacing the remote-viewer `protobuf.js` runtime `unsafe-eval` dependency and adding browser-level E2E coverage for viewer/chat/CDAP UX paths. Those are residual hardening recommendations, not blockers to closing this audit.
